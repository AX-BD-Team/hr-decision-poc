# PoC 시나리오 4종 (Prototype 반영용)

> 목적: Prototype(Decision Workflow Canvas)에 바로 붙여넣어 쓸 수 있도록 **시나리오 정의 + 의사결정 기준 + 데이터 매핑**을 한 파일로 정리  
> 전제: HR 관리 목적 데이터 → **경영 판단(설명 가능)** 데이터로 전환. 정량 정확성은 제한적이며 **룰/단위/정의 고정**이 필수.

---

## 공통 전제 및 데이터 액션 (필수)

### 공통 한계(Prototype에 항상 노출)
- 본 PoC는 **절대값/ROI/개인단가 산출이 목적이 아님**. “왜 그 선택이 합리적인가”를 **근거 연결**로 설명한다.
- 프로젝트 참여 기간/건수 등 일부 지표는 **단위(MM/개월/년) + 집계 규칙(동시투입 중복합산 여부)**이 확정된 범위에서만 해석한다.
- 수행역할/직무는 **표준 코드북으로 정규화한 결과**를 사용(자유텍스트 원문은 참고용).
- 개인식별정보는 **person_key 익명화**(원본 매핑은 별도 보안영역).

### 이번 주 즉시 액션(체크리스트)
1. **프로젝트 참여 기간 단위** 확정(권장: MM 또는 개월) + **중복합산 규칙** 명시  
2. **수행역할 표준화(role_codebook)** + 수행역할(1~3) **Top3 산정 기준** 정의(최근 6~12M/빈도 등)  
3. 사번/성명 → **person_key 익명화**(salt 포함 해시)  
4. 조직명 텍스트 → **org_code(계층)** 매핑(본부/담당/팀 레벨 분리)

## 시나리오 ① TO 추가 요청

### 🔑 핵심 질문
“정말 인력이 부족한가, 아니면 현재 구조와 배치가 문제인가?”

### 🎯 기대 답변(Prototype Output)
- **증원 vs 재배치 vs 구조조정** 중 3개 대안을 제시하고,
- 선택이 합리적인 이유를 **사람·역할·시간(최근 6~12M)·리스크·비용(범위)** 근거로 설명

### 🧭 의사결정 기준
- 업무 과부하 실재성: 특정 개인/역할에 과부하가 집중되는가
- 구조적 대안: 재배치·역할 조정으로 해소 가능한가
- 대체 가능성: 내부 대체 인력 풀은 있는가
- 리스크 감수: 지연 시 사업·고객 리스크 수준
- 비용 합리성: 증원이 다른 대안 대비 타당한가

### 📊 데이터 점검 보정(핵심)
- HR Master 요약만으로는 **“최근 과부하”를 근거로 증명하기 어려움** → 가능하면 `FACT_ASSIGNMENT_12M`에서 파생 지표 생성
- “가동률(정성)”은 참고로만 사용하고, Prototype에서는 **과부하 집중도 지표**를 최소 1개 산출

## 시나리오 ② 상시 조직 변경 / R&R

### 🔑 핵심 질문
“조직을 바꾸는 게 맞나, 아니면 역할과 책임을 바꾸는 게 맞나?”

### 🎯 기대 답변
- 조직개편을 하든 안 하든 **왜 그 선택이 합리적인지 설명 가능한 구조**
- 역할 중복/공백/병목을 근거로, **부분 조정 vs 구조 변경** 대안 비교

### 🧭 의사결정 기준
- 판단 병목: 의사결정이 지연되는 역할/지점
- 역할 중복/공백: 실제 업무 흐름과 조직 구조 불일치
- 변경 효과: 부분 조정으로 충분한가
- 조직 리스크: 변경 시 단기 혼란/저항
- 설명 가능성: “왜 안 바꿨는지”도 설명 가능한가

### 📊 데이터 점검 보정(핵심)
- HR Master의 수행역할(1~3)은 **프로젝트별 실제 역할을 대체할 수 없음**  
  → “업무혼재/중복·공백 탐지”를 하려면 `FACT_ASSIGNMENT_12M`이 사실상 필수
- “역할 중복/공백”은 **판정 룰 + 임계치**를 먼저 고정(정치적 민감도 방어 장치)

## 시나리오 ③ 솔루션 사업화 내부 체계 (확장형 / Phase‑2)

### 🔑 핵심 질문
“이 솔루션을 사업화하려면 내부적으로 어떤 조직·역할·투자가 필요하며, 그 선택은 왜 합리적인가?”

### 🎯 기대 답변
- 사업화 방식(내부/외부/혼합)별로 **조직·인력·비용 구조 차이**를 비교해 판단 근거 제공

### 🧭 의사결정 기준
- 단계성: PoC→Pilot→확산 단계별 조직/역할
- 내부 준비도: 핵심 역할을 내부에서 충족 가능한가
- 투자 리스크: 초기 체계 구축 비용·리스크
- 조직 영향: 기존 조직에 미치는 부작용
- 대안 비교: 내부 구축 vs 외부 협업

### 📊 데이터 점검 보정(핵심)
- HR Master만으로는 **수요(파이프라인)·투자 가정**이 없어서 답이 “추정”이 됨  
  → Phase‑2에서 **기회ID/요구역할/필요MM/착수시점/확도** 최소 구조화가 필요(예: BizForce/VRB)
- 단계별 역할/조직 템플릿은 **기획 산출물(신규 정의 데이터)**

## 시나리오 ④ 역량 강화 (HRD 트랙)

### 🔑 핵심 질문
“조직/인력의 역량을 어떻게 강화할 것인가?”

### 🎯 기대 답변
- 역량 갭을 줄이기 위한 교육·배치·육성 전략(운영/개발 영역)

### 🧭 의사결정 기준
- 역량 갭: 현재–목표 역량 차이
- 개발 수단: 교육/배치/멘토링
- 적용 시점: 단기 판단 영향 여부
- 측정 가능성: 효과 설명 가능성

### 📊 데이터 점검 보정(핵심)
- 교육 효과/성과 연계 데이터가 없으면 “개발 운영” 수준에서 멈춤  
- 메인 PoC(경영 판단 구조화)와 결이 달라 **별도 HRD/People Analytics 트랙**으로 분리 권장

## Prototype 반영 가이드(화면/컴포넌트 기준)

### 1) Scenario Selector(좌측)
- ① TO / ② 조직변경 / ③ 사업화(Phase‑2) / ④ 역량강화(HRD) 토글

### 2) Decision Criteria Panel(상단)
- 각 시나리오의 “의사결정 기준” 5개 항목을 체크리스트로 고정
- 항목별 Evidence 연결률(근거 링크 개수) 표시

### 3) Evidence & Data Readiness Panel(우측)
- 필수 데이터셋(초록), 권장(노랑), 결측(빨강)  
- “단위/룰 미정”은 결측과 동일하게 취급(경영 설득 리스크)

### 4) Output Section(중앙)
- 대안 3개(구성2+대안1) 비교 표준 포맷 유지
- 각 대안에 “근거(Evidence) 링크 / 가정(Assumption) / 한계(Limit)” 3블록 고정
